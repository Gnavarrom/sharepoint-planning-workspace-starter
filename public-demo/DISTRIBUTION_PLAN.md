# Distribution Plan

This file is about how to share the resource publicly and what you can expect from the post.

## Main goal

The goal is not only to show the result.

The goal is to show this message clearly:

- I knew the SharePoint process well
- I knew what I wanted to improve
- I used Codex in VS Code to explore what was possible and build toward that result
- now I am sharing a free resource so others can apply the same pattern

## Best way to publish

### On LinkedIn

Publish:

- one short video recorded from `video/index.html`
- or a carousel using the images in `images/`
- a post built from `LINKEDIN_POST.md`

### For the free resource

Use one of these options:

1. A public GitHub repository with only the sanitized `public-demo/` material
2. A ZIP file shared through Google Drive, OneDrive, or Dropbox
3. A public Notion or simple landing page with a download link to the ZIP

## Recommended option

The cleanest option is:

1. create a new public GitHub repo only for the sanitized starter pack
2. upload the contents of `public-demo/`
3. name it something like `sharepoint-planning-workspace-starter`
4. link that repo in the comments or in your profile

## What people should receive

If someone asks for the free resource, they should get:

- the public-safe demo assets
- the starter guide
- the list template
- the sample CSV files
- the story script

This makes the post useful, not just inspirational.

## What you can get from the post

- credibility as someone who can identify workflow problems and improve them
- visibility from people interested in SharePoint, internal tools, and AI-assisted development
- DMs from people who want the free resource
- new professional connections
- conversations that may lead to consulting, collaboration, or job opportunities

## Suggested CTA

- Comment `starter pack` and I will share the free resource.
- If you are exploring something similar in SharePoint, send me a message.
- I am happy to share the public-safe starter pack with anyone trying to improve an internal workflow.

## Important publishing rule

Do not publish the original project root as-is.

If you want to share code or files publicly, share only a sanitized package or a sanitized repo built from `public-demo/`.
